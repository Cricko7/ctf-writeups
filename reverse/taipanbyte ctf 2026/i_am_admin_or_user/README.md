# CTF Writeup: webserver.tb — «Пользователь я ли обычный или право имею»

## Описание задачи

Нам дан бинарный файл `webserver.tb`, который реализует веб-сервер.  
Из описания: *«Я написал программу, которая позволяет запустить свой веб-сервер! Правда, он не впускает меня самого...»*

Нужно найти способ получить флаг, обойдя проверку прав доступа.

---

## Анализ

### 1. Определение типа файла

```bash
file webserver.tb
```

**Вывод:**
```
ELF 64-bit LSB pie executable, x86-64, dynamically linked, not stripped
```

Это 64-битный ELF-файл для Linux, не удалены символы (упрощает реверс).

### 2. Поиск строк

```bash
strings webserver.tb | grep -i "user\|flag\|admin"
```

**Вывод:**
```
user=user
user=admin
unknown user
```

Сразу видно, что сервер оперирует понятиями «обычный пользователь» (`user=user`) и «администратор» (`user=admin`).  
Также есть строка `unknown user` — ответ при неудачной аутентификации.

### 3. Запуск сервера

```bash
chmod +x webserver.tb
./webserver.tb
```

Сервер слушает порт `8080`:

```
Server is listening on port 8080...
```

### 4. Первые запросы

Пробуем подключиться через `curl`:

```bash
curl -v http://localhost:8080/
```

**Ошибка:**
```
Received HTTP/0.9 when not allowed
```

Это значит, что сервер не возвращает корректный HTTP-ответ, а выдаёт сырые данные.  
Следовательно, это **не полноценный HTTP-сервер**, а просто TCP-сервер, читающий строку и отвечающий текстом.

---

## Реверс-инжиниринг

### 1. Поиск строк в бинарнике

```bash
hexdump -C webserver.tb | grep -i "user"
```

**Вывод:**
```
00002030  3a 20 00 75 73 65 72 3d  75 73 65 72 00 70 65 72  |: .user=user.per|
00002050  75 73 65 72 3d 61 64 6d  69 6e 00 00 00 00 00 00  |user=admin......|
000020a0  2e 00 75 6e 6b 6e 6f 77  6e 20 75 73 65 72 00 00  |..unknown user..|
```

Строки `user=user` и `user=admin` заканчиваются нулевым байтом (как строки C).  
Строка `unknown user` — это ответ сервера при ошибке.

### 2. Проверка работы сервера через `nc`

Пробуем отправить строку `user=admin` с переводом строки:

```bash
echo "user=admin" | nc localhost 8080
```

**Ответ:**
```
unknown user
```

Пробуем `user=user`:

```bash
echo "user=user" | nc localhost 8080
```

**Ответ:**
```
unknown user
```

Пробуем произвольный ввод:

```bash
echo "admin" | nc localhost 8080
```

**Ответ:**
```
unknown user
```

Сервер на **любой** ввод с `\n` отвечает `unknown user`.

### 3. Гипотеза о точном сравнении строк

Скорее всего, сервер сравнивает **в точности** строку из сокета с эталоном `user=admin` или `user=user`, **без учёта символа новой строки**.

Если мы отправляем через `echo`, то добавляется `\n` в конце. Сервер читает строку до `\n`, но при сравнении, возможно, учитывает и этот символ, либо ожидает строку **без завершающего перевода строки**.

### 4. Отправка строки без `\n`

Используем `printf` (не добавляет `\n` по умолчанию):

```bash
printf "user=admin" | nc localhost 8080
```

**Ответ:**
```
Q1RGezRtXzFfNGRtMW5fNzBfbXk1M2xmP30=
```

🎉 Сервер вернул **Base64-строку**!

---

## Декодирование флага

```bash
echo "Q1RGezRtXzFfNGRtMW5fNzBfbXk1M2xmP30=" | base64 -d
```

**Результат:**
```
CTF{4m_1_4dm1n_70_my53lf?}
```

Флаг получен.

---

## Объяснение уязвимости

Сервер реализует простую аутентификацию:

1. Читает данные из сокета до символа `\n`.
2. Сравнивает прочитанную строку с эталоном `user=admin` **побайтово** (функцией `strcmp` или `memcmp`).
3. Если строка совпадает — отправляет закодированный флаг.
4. Если нет — отправляет `unknown user`.

Ошибка в том, что сервер **не удаляет символ перевода строки** из ввода.  
При отправке через `echo "user=admin"` в конце добавляется `\n`, строка становится `user=admin\n`, и сравнение не проходит.

При отправке через `printf "user=admin"` символ `\n` не добавляется, строка точно соответствует эталону, и сервер выдаёт флаг.

## Эксплуатация

```bash
printf "user=admin" | nc localhost 8080 | base64 -d

Вывод:

## Вывод

Задача демонстрирует важность учёта управляющих символов (`\n`, `\r`) при сравнении строк в сетевых сервисах.  
Даже простая проверка может быть обойдена, если не нормализовать входные данные.

Флаг: **`CTF{4m_1_4dm1n_70_my53lf?}`**
